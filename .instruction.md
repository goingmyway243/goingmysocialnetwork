# GoingMy Social Network - AI Agent Instructions

## Project Overview
GoingMy is a modern social network platform built with microservice architecture principles. The project emphasizes scalability, maintainability, and contemporary UI/UX design patterns inspired by Apple's glassmorphism aesthetic.

## Tech Stack

### Backend
- **Framework**: .NET 10
- **Architecture**: Microservices`
- **Development**: .NET Aspire for local development orchestration
- **Patterns**: MediatR for CQRS, Repository pattern for data access, Dependency Injection for service management
- **Services**:
  - GoingMy.AuthService - Authentication and authorization
  - GoingMy.PostService - Post management and social interactions
  - GoingMy.ServiceDefaults - Shared service configurations and extensions
  - GoingMy.AppHost - Aspire application host for service orchestration

### Frontend
- **Framework**: Angular 20
- **UI Library**: PrimeNG
- **State Management**: Angular Signals (modern reactive approach)
- **Code Organization**: Code Flow Blocks pattern for structured, maintainable code
- **Design System**: Apple glassmorphism aesthetic with transparency, blur effects, and modern spacing

## Architecture Principles

### Microservices
- Each service maintains its own database and API contracts
- Services communicate through well-defined HTTP APIs
- Authentication flows through the centralized AuthService
- All services run in containers for consistency

### CQRS Pattern Implementation
The PostService (and future services) follow the CQRS (Command Query Responsibility Segregation) pattern using MediatR:

**Commands** (`Application/Commands/` directory)
- Each command file contains both the command record and its handler class
- Example: `CreatePostCommand.cs` contains `CreatePostCommand` record + `CreatePostCommandHandler` class
- Commands represent write operations (create, update, delete)
- Handlers receive repository injections and execute business logic
- File naming: `[ActionName]Command.cs` (e.g., `CreatePostCommand.cs`, `UpdatePostCommand.cs`, `DeletePostCommand.cs`)
- Handler naming: `[CommandName]Handler` (e.g., `CreatePostCommandHandler`)

**Queries** (`Application/Queries/` directory)
- Each query file contains both the query record and its handler class
- Example: `GetPostsQuery.cs` contains `GetPostsQuery` record + `GetPostsQueryHandler` class
- Queries represent read operations (fetch, search)
- Handlers perform data retrieval and mapping to DTOs
- File naming: `Get[Resource][Criteria]Query.cs` (e.g., `GetPostsQuery.cs`, `GetPostByIdQuery.cs`)
- Handler naming: `[QueryName]Handler` (e.g., `GetPostsQueryHandler`)

**Controllers** (`API/Controllers/` directory)
- REST controllers handle HTTP requests and delegate to MediatR
- Single responsibility: receive request, extract claims, invoke command/query, return response
- Use request DTOs (different from domain commands) for API contract
- Example: `PostsController` dispatches to `CreatePostCommand`, `UpdatePostCommand`, etc.
- Proper HTTP status codes and error handling via ProducesResponseType attributes

**Application Layer Structure**
```
ServiceName.Application/
├── Commands/
│   ├── CreatePostCommand.cs       # CreatePostCommand record + CreatePostCommandHandler
│   ├── UpdatePostCommand.cs       # UpdatePostCommand record + UpdatePostCommandHandler
│   └── DeletePostCommand.cs       # DeletePostCommand record + DeletePostCommandHandler
├── Queries/
│   ├── GetPostsQuery.cs           # GetPostsQuery record + GetPostsQueryHandler
│   └── GetPostByIdQuery.cs        # GetPostByIdQuery record + GetPostByIdQueryHandler
└── Dtos/
    └── PostDto.cs                 # Data Transfer Objects for API responses
```

**Adding New Commands/Queries**
1. Create file in appropriate directory: `Commands/[ActionName]Command.cs` or `Queries/[ActionName]Query.cs`
2. Define the command/query record with `IRequest<TResponse>` interface
3. Define the handler class with `IRequestHandler<TCommand/Query, TResponse>` interface
4. Inject `IPostRepository` or other dependencies in handler constructor
5. Implement `Handle()` method with business logic
6. In controller, create new action method that calls `mediator.Send(new YourCommand(...))`
7. MediatR automatically discovers and registers handlers via assembly scanning in `Program.cs`

## Frontend Development Guidelines

### Signals & Reactive Programming
- Use Angular Signals for state management instead of RxJS where appropriate
- Leverage computed signals for derived state
- Implement effect for side effects in component lifecycle
- Keep signal granularity at feature/component level

### Code Flow Blocks
- Organize component logic into discrete, testable code blocks
- Each block handles a single responsibility (display, interaction, validation, etc.)
- Clear separation between template, component class, and services
- Improves readability and maintainability

### PrimeNG Integration
- Use PrimeNG components for consistent UI patterns
- Customize components using glassmorphism design tokens
- Leverage PrimeNG's theming system for brand consistency
- Document custom theme overrides in component stylesheets

### Glassmorphism Design
- Apply backdrop blur effects for depth and hierarchy
- Use semi-transparent backgrounds with rgba values
- Implement proper contrast ratios for accessibility
- Reference Apple's Human Interface Guidelines for spacing and typography

## Code Standards

### .NET
- Follow Microsoft .NET naming conventions
- Use nullable reference types (`#nullable enable`)
- Implement dependency injection through constructor
- Apply async/await for I/O operations
- Write unit and integration tests for business logic

### CQRS Pattern (MediatR)
- **Commands**: Contains write operations (create, update, delete)
  - Use `IRequest<TResponse>` interface for type safety
  - Consolidate command and handler in the same file
  - Name command handlers as `[CommandName]Handler`
  - Validate ownership before modifying resources (throw `UnauthorizedAccessException`)
  - Throw `InvalidOperationException` for not-found resources
- **Queries**: Contains read operations (fetch, search, list)
  - Use `IRequest<TResponse>` interface, typically returning DTOs or collections
  - Consolidate query and handler in the same file
  - Name query handlers as `[QueryName]Handler`
  - Return `null` or empty collections appropriately for missing resources
- **Controllers**: REST endpoints that dispatch to commands/queries
  - Single responsibility: HTTP request → claim extraction → MediatR dispatch → HTTP response
  - Use separate request DTOs from domain commands for API contracts
  - Include `ProducesResponseType` attributes for OpenAPI documentation
  - Catch specific exceptions and map to appropriate HTTP status codes
- **DTOs**: Data transfer objects in Application/Dtos/
  - Keep minimal and focused on what the API needs to return
  - Use records for immutability
  - Separate from domain entities to maintain architectural boundaries

### Angular
- Use strict TypeScript mode
- Implement OnDestroy for cleanup in signals/subscriptions
- Follow Angular style guide for component organization
- Use standalone components where appropriate
- Document complex signal flows with comments
