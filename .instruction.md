# GoingMy Social Network - AI Agent Instructions

## Project Overview
GoingMy is a modern social network platform built with microservice architecture principles. The project emphasizes scalability, maintainability, and contemporary UI/UX design patterns inspired by Apple's glassmorphism aesthetic.

## Tech Stack

### Backend
- **Framework**: .NET 10
- **Architecture**: Microservices
- **Development**: .NET Aspire for local development orchestration
- **Services**:
  - GoingMy.AuthService - Authentication and authorization
  - GoingMy.PostService - Post management and social interactions
  - GoingMy.ServiceDefaults - Shared service configurations and extensions
  - GoingMy.AppHost - Aspire application host for service orchestration

### Frontend
- **Framework**: Angular 20
- **UI Library**: PrimeNG
- **State Management**: Angular Signals (modern reactive approach)
- **Code Organization**: Code Flow Blocks pattern for structured, maintainable code
- **Design System**: Apple glassmorphism aesthetic with transparency, blur effects, and modern spacing

## Architecture Principles

### Microservices
- Each service maintains its own database and API contracts
- Services communicate through well-defined HTTP APIs
- Authentication flows through the centralized AuthService
- All services run in containers for consistency

### Backend Structure (Per Service except AppHost, AuthService, ServiceDefaults)
```
ServiceName/
├── src/
│   ├── ServiceName.API/           # API layer, controllers, HTTP endpoints
│   ├── ServiceName.Application/   # Business logic, use cases
│   ├── ServiceName.Domain/        # Domain entities, value objects
│   └── ServiceName.Infrastructure/ # Data access, external integrations
└── tests/
    ├── ServiceName.Tests/         # Unit tests
    └── ServiceName.IntegrationTests/ # Integration tests
```

### Development Experience
- Use .NET Aspire to run all services locally with proper dependency management
- `AppHost.cs` orchestrates service startup and configuration
- All services integrate health checks and proper logging
- Development settings in `appsettings.Development.json` for local overrides

## Frontend Development Guidelines

### Signals & Reactive Programming
- Use Angular Signals for state management instead of RxJS where appropriate
- Leverage computed signals for derived state
- Implement effect for side effects in component lifecycle
- Keep signal granularity at feature/component level

### Code Flow Blocks
- Organize component logic into discrete, testable code blocks
- Each block handles a single responsibility (display, interaction, validation, etc.)
- Clear separation between template, component class, and services
- Improves readability and maintainability

### PrimeNG Integration
- Use PrimeNG components for consistent UI patterns
- Customize components using glassmorphism design tokens
- Leverage PrimeNG's theming system for brand consistency
- Document custom theme overrides in component stylesheets

### Glassmorphism Design
- Apply backdrop blur effects for depth and hierarchy
- Use semi-transparent backgrounds with rgba values
- Implement proper contrast ratios for accessibility
- Reference Apple's Human Interface Guidelines for spacing and typography

## Service APIs

### Authentication Service
- **Port**: Configured in Aspire
- **Key Endpoints**: Register, Login, Validate, Refresh Token
- **Responsibility**: Identity management, JWT token generation, OAuth integrations

### Post Service
- **Port**: Configured in Aspire
- **Key Endpoints**: Create Post, Get Posts, Like/Comment, Delete Post
- **Responsibility**: Social content management, interactions, feed generation

## Development Workflow

1. **Local Setup**:
   - Open `GoingMy.Services.slnx` at workspace root
   - Run AppHost to orchestrate all services
   - Check Aspire dashboard for service health

### Pre-Build Verification
Before requesting or building any project:
   - **Check all using statements** in modified files are correct and necessary
   - Remove unused imports to keep code clean
   - Ensure all referenced namespaces exist in the project dependencies
   - Verify NuGet packages are properly referenced in `.csproj` files
   - Run `dotnet build` to catch missing using statements early

2. **Backend Changes**:
   - Make changes in appropriate service layers
   - Run tests: `dotnet test` in service directory
   - Follow SOLID principles and clean architecture

3. **Frontend Changes**:
   - Edit Angular components in appropriate feature modules
   - Use Angular CLI for scaffolding: `ng generate component`
   - Apply glassmorphism styles to new components
   - Test signals reactivity and code flow blocks

4. **Service Communication**:
   - Services communicate via HTTP/REST
   - Authentication passes JWT tokens in Authorization headers
   - Document API contracts in `.http` files (e.g., `GoingMy.Auth.API.http`)

## Code Standards

### .NET
- Follow Microsoft .NET naming conventions
- Use nullable reference types (`#nullable enable`)
- Implement dependency injection through constructor
- Apply async/await for I/O operations
- Write unit and integration tests for business logic

### Angular
- Use strict TypeScript mode
- Implement OnDestroy for cleanup in signals/subscriptions
- Follow Angular style guide for component organization
- Use standalone components where appropriate
- Document complex signal flows with comments

## Important Notes
- All local development must use .NET Aspire for consistency
- Frontend and Backend are developed in parallel with clear API contracts
- Glassmorphism is not just visual - it represents the project's modern, elegant approach
- PrimeNG components should be customized, not replaced
- Maintain separation of concerns across all service layers
