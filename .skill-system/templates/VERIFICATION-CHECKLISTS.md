# Verification Checklist Templates\n\nReusable checklists for common verification scenarios.\n\n## Backend Code Review Checklist\n\n```markdown\n## Code Review Checklist\n\n### Correctness\n- [ ] Logic implements requirements correctly\n- [ ] No edge cases missed\n- [ ] Error handling comprehensive\n- [ ] No N+1 database queries\n- [ ] Async/await used appropriately\n\n### Completeness\n- [ ] Unit tests present (>80% coverage)\n- [ ] Integration tests for data access\n- [ ] Public APIs documented (XML comments)\n- [ ] Configuration externalized\n\n### Code Quality\n- [ ] SOLID principles followed\n- [ ] DRY principle respected\n- [ ] No magic numbers/strings\n- [ ] Method names descriptive\n- [ ] Classes have single responsibility\n\n### Security\n- [ ] Input validation present\n- [ ] SQL injection prevented\n- [ ] No secrets in code\n- [ ] Authorization checks present\n- [ ] Sensitive data not logged\n\n### Performance\n- [ ] No blocking calls (.Result, .Wait())\n- [ ] Query optimization checked\n- [ ] Memory leaks prevented (IDisposable)\n- [ ] Caching considered\n\n### Decision: APPROVE / REQUEST CHANGES / REJECT\n```\n\n## Frontend Code Review Checklist\n\n```markdown\n## Angular Component Review\n\n### Component Design\n- [ ] Smart/dumb separation clear\n- [ ] @Input/@Output properly defined\n- [ ] Change detection optimized (OnPush)\n- [ ] Lifecycle hooks used correctly\n\n### Reactive Programming\n- [ ] Observables used (not subscriptions)\n- [ ] Async pipe used in templates\n- [ ] Proper unsubscription (takeUntil)\n- [ ] No memory leaks\n\n### Styling\n- [ ] Uses design tokens (no hard-coded colors)\n- [ ] Glassmorphism applied correctly\n- [ ] Responsive design verified\n- [ ] Dark mode supported\n\n### Testing\n- [ ] Unit tests present\n- [ ] Edge cases covered\n- [ ] Coverage > 80%\n- [ ] Component interaction tested\n\n### Decision: APPROVE / REQUEST CHANGES / REJECT\n```\n\n## API Contract Review Checklist\n\n```markdown\n## API Contract Verification\n\n### REST Conventions\n- [ ] Resource-oriented (nouns, not verbs)\n- [ ] HTTP methods correct (GET, POST, PUT, DELETE)\n- [ ] Status codes appropriate (200, 201, 400, 404, 500)\n- [ ] URL versioning (/api/v1/...)\n\n### Request/Response\n- [ ] Request validation present\n- [ ] Response format consistent\n- [ ] Error response standardized\n- [ ] Documentation complete\n\n### Data Contracts\n- [ ] DTOs properly structured\n- [ ] Null handling defined\n- [ ] Date format standardized (ISO 8601)\n- [ ] No sensitive data exposed\n\n### Security\n- [ ] Authentication required\n- [ ] Authorization enforced\n- [ ] Rate limiting configured\n- [ ] Input sanitized\n\n### Decision: APPROVE / REQUEST CHANGES / REJECT\n```\n\n## Design Review Checklist\n\n```markdown\n## UI/UX Design Review\n\n### Glassmorphism Consistency  \n- [ ] Glass effect properly applied\n- [ ] Transparency opacity correct (0.8)\n- [ Dark mode variants exist\n\n### Responsiveness\n- [ ] Mobile layout (320px) works\n- [ ] Tablet layout (768px) works\n- [ ] Desktop layout (1200px) works\n- [ ] Touch targets > 44x44px\n\n### CHANGES / REJECT\n```\n\n## Pre-Deployment Checklist\n\n```markdown\n## Deployment Checklist\n\n### Code Quality\n- [ ] All tests pass\n- [ ] Code review approved\n- [ ] SonarQube quality gate
